A strict static analyzer for Go.

Designed for code written by humans and LLMs. Catches the bug classes that compile, pass tests, and ship anyway. Zero external linter dependencies — every rule is implemented from scratch on top of go/ast and go/types.

The other 4,153 — strong opinions, often bugs

These are conventions gox enforces. Each one is opt-out with a one-line annotation, but every opt-out forces you to write down why — the reason becomes documentation.

1,254 noglobals Mutable package-level var. Process-global state. Annotate with // global-ok: <why> if intentional.
1,063 namedargs Two adjacent params of the same basic type (string/int/bool) called without /* paramName */ at the call site. Catches transfer(orderID, userID)-vs-transfer(userID, orderID) bugs that compile fine. Most-frequent param names hit: reasons, agencyID, agentID, carrier, method, state.
816 shadow A := that re-declares an outer variable. Almost all of these shadow err in nested scopes — the classic "the outer error never gets returned" bug.
694 banany any / interface{} in declarations. Annotate with // any-ok: <why> for boundary cases.
236 errcheck Error returns silently dropped. bytes.Buffer.WriteString, strings.Builder.WriteString, and fmt.Print* are pre-exempted (docs-guaranteed nil error).
90 forcetypeassert Type assertions without the comma-ok form — v.(T) panics on mismatch.

Why it exists

Most bugs an LLM introduces are not syntactic. The compiler can't see them, your unit tests don't trigger them, and they hide in the codebase for months. Three examples:

  • transfer(orderID, userID) — passes type-check; silently transfers from the wrong account.
  • err := process(); if err != nil { err := log(err); ... } — the outer err is shadowed; the return value is always nil.
  • Add a new CarrierStatus variant; nine switch statements quietly fall through to default.

Every rule was chosen because we have seen the bug it prevents ship to production, in this codebase or one nearby. The defaults are loud on purpose: fail closed, opt out explicitly, write down why. The "why" is the documentation.

Try it

One install. Two commands. Three minutes.

# install the binary
go install github.com/mentasystems/gox/cmd/gox@latest

# run it on any Go project — same syntax as `go build`
cd ~/your-go-project
gox check ./...

# wire it into Claude Code as a PostToolUse hook
# (runs after every Edit/Write/MultiEdit on .go files)
gox install claude

Cold scan of mono: 9.4s. Warm: 2.6s. The cache is per-package, mtime-keyed, lives in ~/.cache/gox/v2.

How opt-outs work. Every rule has one comment annotation that turns it off for that specific line / declaration / file. You must include a reason — empty reasons are ignored.

// safe-ignore: this fire-and-forget is intentional, channel closed by sender

// global-ok: spanner client is a process-level singleton by design

// any-ok: protobuf reflection requires interface{} here

// exhaustive-ok: default catches all future variants on purpose

// goroutine-ok: lifecycle bounded by HTTP request, ctx already cancellable

Adopting it without a PR storm

4,201 issues is a lot. The intended workflow is not to fix them all at once.

  1. Run gox install claude on your laptop today.
  2. From now on, every Go file Claude edits gets gox-checked. Claude sees the issues and fixes (or annotates) them before continuing.
  3. New code stays clean. Old code stays as it is.
  4. Eventually the diff between "untouched legacy" and "Claude-touched, gox-clean" shrinks by itself.
